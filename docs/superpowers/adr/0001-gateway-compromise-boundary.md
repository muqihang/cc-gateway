# ADR 0001: Gateway Compromise Boundary

- Status: accepted for Phase 0
- Decision: `protected_gateway`
- Scope: formal-pool Sub2API deployment and oracle-lab governance

## Context

Sidecar authentication alone cannot distinguish an authorized request from a malicious-but-valid request generated by a compromised Gateway when the Gateway holds final sidecar authority. Formal-pool deployments therefore need an explicit threat-model choice. An omitted or unknown choice is unsafe and must fail closed.

## Options evaluated

### `protected_gateway`

Gateway-process compromise remains in scope. The Gateway must not hold final sidecar authorization authority. An independently isolated policy broker must issue a short-lived, single-request capability binding the account, credential and proxy generations, persona, transport profile, target, method, route, final headers and body, attempt ID, deadline, response policy, key epoch, and nonce. The sidecar independently verifies the capability and current manifest authority.

This option preserves the intended compromise boundary but the Phase 4 policy broker is not yet locally verified. Phase 0 therefore permits `protected_gateway` only for `preflight` and `local-capture`. Production, real-canary, and any other upstream mode fail closed with the exact code `protected_gateway_authority_unavailable`.

### `trusted_gateway`

Gateway-process compromise is explicitly out of scope. This is the only Phase 0 production-compatible option, but a compromised Gateway is outside the sidecar prevention boundary. Sidecar authentication can still protect against external, loopback-peer, and same-container unprivileged attackers; it cannot prevent malicious-but-valid requests generated by the Gateway. Release claims and telemetry must state this limitation. Incident response prioritizes Gateway isolation, secret rotation, manifest revocation, and forensic preservation.

## Decision

Select `protected_gateway` for Phase 0. Formal-pool configuration must explicitly declare `shared_pool.gateway_compromise_boundary: protected_gateway`; no implicit default is introduced. The normalized choice is safe to record in run manifests and generated evidence.

Production and real-canary remain disabled until the Phase 4 policy broker is locally verified. This deferred production capability is a mandatory later gate, not a blocker for Phase 0 governance exit while those capabilities remain disabled.

Operators who require Phase 0 production compatibility may explicitly select `trusted_gateway`, accepting and documenting its narrower threat model. No deployment or security claim may imply protection against a compromised Gateway under that choice.

## Consequences

- Missing and unknown declarations fail closed.
- `protected_gateway` local capture and preflight remain available for Phase 0 evidence work.
- `protected_gateway` production and real-canary fail with `protected_gateway_authority_unavailable` until Phase 4 broker verification.
- `trusted_gateway` is production-compatible in Phase 0 only with the compromised-Gateway limitation explicitly accepted.
- Run manifests, generated evidence, operational docs, and release approval must carry the normalized declared choice.
